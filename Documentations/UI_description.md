## User Interface Design

The system provides multiple interfaces designed for different user roles interacting with the log-driven incident response platform.

---

### 1. Graphical User Interface (GUI) Dashboard

**Purpose**

The GUI dashboard is primarily used for **incident monitoring and management**. It allows users to visualize the operational status of the system and track incidents generated by the detection service.

**Supported Components**

- Incident Management Service
- Detection Service
- Notification Service
- Reporting & Audit Logging

**Key Functions**

- Monitor active and historical incidents
- View incident severity and affected services
- Observe alerts generated by the detection engine
- Track automated responses and system actions
- Generate reports and audit logs

**Users**

- Security Analysts  
- System Operators

**Justification**

A GUI dashboard enables users to **quickly understand system state through visual representations** such as tables, alerts, and dashboards. This is essential for real-time monitoring of incidents and system activity.

---

### 2. Command Line Interface (CLI)

**Purpose**

The CLI is used by **security analysts** when deeper investigation or manual intervention is required after an incident is detected.

**Supported Components**

- Log Repository
- Incident Store
- Detection Service
- Response Actuation

**Key Functions**

- Query and filter logs from the log repository
- Inspect incident details and correlation results
- Trigger manual response actions
- Restart services or execute remediation commands
- Perform debugging and system diagnostics

**Users**

- Security Analysts  
- Incident Response Engineers

**Justification**

The CLI provides **high flexibility and efficiency**, allowing experienced users to perform detailed investigations and execute system actions quickly using commands or scripts.

---

### 3. Direct Manipulation Interface (DMI)

**Purpose**

The DMI is used by **system administrators** to create, modify, and manage automated response playbooks that are stored in the playbook repository.

**Supported Components**

- Playbook Store
- Playbook Selection & Execution Module
- Response Actuation

**Key Functions**

- Visually design response workflows
- Drag and arrange response actions (restart, rollback, notify)
- Modify existing playbooks
- Define execution conditions and dependencies

**Users**

- System Administrators

**Justification**

A Direct Manipulation Interface allows administrators to **visually configure response workflows**, making complex automation easier to design and maintain without requiring manual scripting.

---

### Conclusion

The system integrates three complementary interfaces:

- **GUI Dashboard** for monitoring and incident management  
- **CLI** for investigation and operational control  
- **DMI** for designing and managing automated response playbooks  

This multi-interface approach ensures that the system effectively supports **monitoring, investigation, and automated response configuration** across different user roles.
